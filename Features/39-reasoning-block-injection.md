# 39 — Reasoning Block Injection

## Summary

Allow users to harvest thinking/reasoning blocks from one assistant node and inject them into another node's context. When the injected block's provider matches the active provider, the block is sent in its native format (Anthropic thinking block with signature, OpenAI encrypted reasoning item). When providers don't match or native format is unavailable, an optional plaintext fallback prepends the reasoning text to the assistant message content in the API payload.

This feature is motivated by empirical research (`../thinking-testing-main/RESEARCH_ANALYSIS.md`) demonstrating that:
- Anthropic thinking blocks in prior assistant turns are visible to the model and steer subsequent responses (100% reproducible, Finding 2/5/8)
- Thinking blocks can be inserted into turns that never had them, causing the model to treat injected content as its own reasoning (Finding 9)
- Signatures verify content authenticity but not conversation context — blocks are portable across conversations (Finding 1)
- OpenAI encrypted reasoning also steers when positioned in the last assistant turn (Finding 13/14)
- Tool results dominate over injected reasoning, providing a partial mitigating factor (Finding 3)
- Safety systems hold regardless of injected thinking (Finding 7)

## Data Model

### ThinkingBlock

New interface replacing the current `thinking?: string` on TreeNode:

```typescript
interface ThinkingBlock {
  id: string;                  // unique block ID (UUID)
  text: string;                // display text (Anthropic: cleartext thinking, OpenAI: summary or "[encrypted]")
  providerId: string;          // 'anthropic' | 'openai' — which provider generated this
  signature?: string;          // Anthropic: base64 signature from thinking content block
  encryptedContent?: string;   // OpenAI: encrypted_content blob from reasoning item
  sourceNodeId?: string;       // provenance: which node this was harvested from (null if original)
  sourceConversationId?: string; // provenance: conversation of source node
  isOriginal: boolean;         // true if generated by the model for this node, false if injected
  plaintextEnabled: boolean;   // if true, use plaintext fallback when native format unavailable
}
```

### TreeNode changes

```typescript
// Remove:
thinking?: string;

// Add:
thinkingBlocks?: ThinkingBlock[];
```

**Migration**: Existing nodes with `thinking: string` are migrated to `thinkingBlocks: [{ id: uuid(), text: thinking, providerId: 'anthropic', isOriginal: true, plaintextEnabled: false }]`. The signature is not available for historical nodes (it was discarded during streaming), so these blocks cannot be replayed natively — only displayed or used as plaintext.

### AppSettings additions

```typescript
reasoningInjectionPlaintextPrefix: string;  // default: '[Prior reasoning: '
reasoningInjectionPlaintextSuffix: string;  // default: ']'
```

### Dexie Migration — Version 8

No index changes needed. `thinkingBlocks` is stored as a non-indexed field on nodes. The migration converts existing `thinking` strings:

```typescript
this.version(8).stores({
  // ... existing tables unchanged — no index changes
}).upgrade(tx => {
  return tx.table('nodes').toCollection().modify(node => {
    if (node.thinking && !node.thinkingBlocks) {
      node.thinkingBlocks = [{
        id: crypto.randomUUID(),
        text: node.thinking,
        providerId: 'anthropic',  // only Anthropic had supportsThinking=true
        isOriginal: true,
        plaintextEnabled: false,
      }];
    }
    delete node.thinking;
  });
});
```

**Note**: Version 8 was previously reserved for RAG embeddings (Feature 19). Since Feature 19 is not started, this version is reassigned. RAG embeddings move to Version 9.

## Capturing Thinking Blocks During Streaming

### Current behavior

The Anthropic provider extracts thinking text from `type: 'thinking'` content blocks but **discards the signature**. The thinking text is passed via `onThinkingComplete(text)` callback and stored as `node.thinking`.

### Required changes (Phase A)

Modify `AnthropicProvider.sendMessage` to:
1. Extract both `thinking` text and `signature` from thinking content blocks
2. Pass them via an enhanced callback: `onThinkingComplete(thinkingBlock: { text: string; signature: string })`
3. `useStreamingResponse` converts this into a full `ThinkingBlock` with `providerId: 'anthropic'`, `isOriginal: true`
4. `finalizeNode` stores the block in `node.thinkingBlocks`

### Required changes (Phase B)

Modify `OpenAIProvider.sendMessage` to:
1. Switch from Chat Completions to Responses API (see ADR-020)
2. Extract `encrypted_content` from reasoning items in the response
3. Generate a text summary (or use "[Encrypted reasoning — X tokens]" placeholder)
4. Pass via the same `onThinkingComplete` callback
5. Store as `ThinkingBlock` with `providerId: 'openai'`, `encryptedContent: blob`

## Copy/Paste UX

### Store state

Add to `useTreeStore`:

```typescript
reasoningClipboard: ThinkingBlock | null;

copyReasoningBlock: (nodeId: string, blockId: string) => void;
pasteReasoningBlock: (targetNodeId: string) => Promise<void>;
clearReasoningClipboard: () => void;
```

### Copy flow

1. User views an assistant node's thinking block in NodeDetailPanel or ThreadMessage
2. Clicks "Copy Reasoning" button on the block
3. Store sets `reasoningClipboard` to a copy of the `ThinkingBlock` with `sourceNodeId` and `sourceConversationId` populated
4. Visual feedback: toast or clipboard indicator

### Paste flow

1. User navigates to a different assistant node
2. In NodeDetailPanel's "Reasoning Blocks" section, clicks "Paste Reasoning"
3. Store creates a new `ThinkingBlock` from the clipboard:
   - `id`: new UUID
   - `isOriginal: false`
   - `sourceNodeId`, `sourceConversationId`: from clipboard
   - All other fields (text, signature, encryptedContent, providerId): copied from clipboard
   - `plaintextEnabled: false` (user can toggle later)
4. Block is appended to `targetNode.thinkingBlocks`
5. Node is persisted to Dexie

### Constraints

- Cannot paste onto the same node the block was copied from (no-op with toast warning)
- Multiple blocks per node are allowed (e.g., one Anthropic + one OpenAI + one edited plaintext)
- Pasting a block with the same `sourceNodeId` as an existing block on the target shows a confirmation: "This node already has a reasoning block from that source. Replace or add another?"

## Context Assembly

### Provider-matching logic

When building messages for an API call from node X:

1. Walk path from X to root, collecting nodes
2. For each assistant node with `thinkingBlocks`:
   a. Find blocks where `block.providerId === activeProviderId` and `block.isOriginal || !block.isOriginal` (all blocks for the matching provider)
   b. For matching blocks: include in native format (Anthropic: `type: 'thinking'` content block with signature; OpenAI: reasoning item with `encrypted_content`)
   c. For non-matching blocks where `block.plaintextEnabled === true`: include as plaintext prepended to the assistant message content
   d. For non-matching blocks where `block.plaintextEnabled === false`: skip entirely

### Anthropic native format

```typescript
// Assistant message with thinking block in context
{
  role: 'assistant',
  content: [
    { type: 'thinking', thinking: block.text, signature: block.signature },
    { type: 'text', text: node.content }
  ]
}
```

If the block has no signature (legacy migrated data), fall back to plaintext if `plaintextEnabled`, otherwise skip.

### OpenAI native format (Phase B)

```typescript
// Reasoning item preceding the assistant message in Responses API input
{ type: 'reasoning', id: generatedId, encrypted_content: block.encryptedContent }
```

**Important**: OpenAI silently discards reasoning from all but the last assistant turn. Show a warning when the user pastes an OpenAI reasoning block onto a non-leaf assistant node.

### Plaintext fallback format

```typescript
// Assistant message content with plaintext reasoning prepended
{
  role: 'assistant',
  content: `${settings.reasoningInjectionPlaintextPrefix}${block.text}${settings.reasoningInjectionPlaintextSuffix}\n\n${node.content}`
}
```

This is computed **at send-time only** — the stored node content is never modified. The context builder applies this transformation when assembling the message array.

### Automatic plaintext fallback

When a node has thinking blocks but the active provider doesn't match any of them:
- Blocks with `plaintextEnabled: true` automatically fall back to plaintext
- The UI shows a subtle indicator on the node: "Reasoning sent as plaintext (provider mismatch)"
- Blocks with `plaintextEnabled: false` are silently skipped

## Edge Cases

- [ ] **Node deletion**: Deleting a source node does not affect injected blocks on other nodes (blocks are copies, not references)
- [ ] **Conversation export/import**: ThinkingBlocks should be included in JSON export. Signatures/encrypted content are preserved but may not be valid if imported into a different account.
- [ ] **Streaming interruption**: If streaming is cancelled mid-thinking, the partial thinking text is captured but the signature may be incomplete/missing. Mark such blocks with `signature: undefined`.
- [ ] **Multiple providers on same node**: A node could have both an Anthropic and OpenAI block. Context builder selects the one matching the active provider.
- [ ] **Provider cascade interaction**: The active provider is resolved at the leaf node (walking up the cascade). All thinking blocks in the path are evaluated against this resolved provider.
- [ ] **Merge nodes**: Merge creates synthetic nodes. Thinking blocks on source nodes are not automatically carried into the merge — the merge content builder handles context separately.
- [ ] **Summarize nodes**: Summary nodes don't carry thinking blocks from their source branches.
- [ ] **Batch execution** (Feature 29): Each batch run uses its own context path. Thinking blocks are included per normal context assembly rules.
- [ ] **Model change mid-branch**: If user switches from Anthropic to OpenAI mid-branch, Anthropic thinking blocks earlier in the path are either sent as plaintext (if enabled) or skipped. No error — silent degradation.
- [ ] **Elicitation agent**: The elicitation agent's context assembly (`agents/contextAssembly.ts`) needs to be updated to handle thinking blocks in the same way as the main chat context builder.
